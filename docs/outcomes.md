# Outcomes Panel

> Audience: Frontend developers, full-stack engineers
> See also: [`workflow.md`](./workflow.md), [`artifact.md`](./artifact.md)

The Outcomes panel (`/outcomes`) acts as the transient, thread-scoped companion to the chatbot. It is responsible for rendering all complex, visual, or structural outputs produced by the agent during a conversation.

While the chat interface handles conversational text, the Outcomes panel presents the tangible *results* (outcomes) of the agent's actions—such as web search results, data visualizations, and in the future, HTML pages and presentations.

---

## 1. Core Architecture

### 1.1 The Role of the Outcomes Panel
- **Thread-Scoped & Ephemeral**: Outcomes live in the `outcome-store.ts` (Zustand) and belong to the current chat session. They are cleared when switching threads.
- **Visual Presentation**: Outcomes render as collapsible, enlargeable cards (`OutcomeCard`) within a grid layout.
- **Bridge to Persistence**: Outcomes are temporary until the user clicks "Save", which persists the outcome to the `artifact` database table (making it available in the persistent Artifacts library).

### 1.2 Block-Based Rendering
An Outcome is composed of visual primitives called "blocks". This allows a single outcome to combine different types of content. Current supported blocks include:
- **CardListBlock**: Clickable cards with thumbnails, excerpts, and citations. Used heavily for:
  - Web search results
  - Knowledge Base (KB) retrievals
  - File lookups
  - SQL data rows
- **ChartBlock**: ECharts visualizations generated by the agent.
- **TextBlock**: Markdown content.

## 2. Detection & Rendering Pipeline

When the agent uses a tool that produces visual output (e.g., searching the web or generating a chart), the system routes that output to the Outcomes panel.

```text
Agent (Tool Call)
    │  e.g., generate_echarts_config() or web_search()
    ▼
PersistingAgent
    │  Persists the tool_call_chunk to entity_run_event
    ▼
SSE Stream → Browser
    │
    ├─► Outcome Store (Zustand)
    │     • Adds new outcome or merges updates
    │     • Maintains block order and visual state
    │
    ├─► OutcomesPanel (/outcomes)
    │     • Renders OutcomeCards based on block types
    │     • Provides Enlarge / Collapse / Save UI
    │
    └─► Chat Message Renderer
          • Replaces raw JSON tool outputs with concise UI references
```

---

## 3. Data Flow — Save to Artifact

When a user identifies a valuable outcome (e.g., a perfect chart or a crucial set of search results), they can save it.

1. **User Action**: Clicks "Save" on an `OutcomeCard`.
2. **API Call**: `POST /api/artifacts/save` is triggered via `useSaveOutcome`.
3. **Persistence**: The server coalesces the underlying `entity_run_event` tool calls, builds a workflow specification, and inserts rows into the `artifact` and `workflow` tables.
4. **State Update**: The in-memory outcome is updated with a `savedArtifactId`, visually indicating it is now permanently stored.

When a user revisits an old thread, the Outcomes panel automatically hydrates (`loadForThread`) by replaying the persisted `entity_run_event` rows.

---

## 4. Future Plans

The Outcomes panel is designed to be extensible. Planned additions include:
- **HTML/UI Rendering**: Displaying agent-generated web pages (via iframe or sandboxed bridges).
- **Presentation (PPT)**: Slide deck previews.
- **Code Displays**: Enhanced syntax-highlighted code editors for generated scripts.
- **Dashboard Construction**: The ability to select multiple saved outcomes and arrange them into persistent dashboards.
- **In-place Editing**: Allowing users to manipulate or edit the content of an outcome directly in the panel.
